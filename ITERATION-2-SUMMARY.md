# Local Comet — Итерация 2: Сводка изменений

## Версия
`v0.2.0` — дата: 2026-03-30

## Новые возможности

### 1. Порт 5051 (конфигурируемый)
- Сервер запускается на **порту 5051** по умолчанию
- Порт конфигурируется через переменные окружения: `LOCAL_COMET_PORT` → `PORT` → `5051`
- Клиентский queryClient обновлён для корректного проксирования через `__PORT_5051__`

### 2. Живой агентный цикл observe → reason → act → re-observe
- Агент работает в цикле: **наблюдение** → **анализ** → **действие** → **повторное наблюдение**
- После каждого действия агент получает новое наблюдение о состоянии страницы (URL, заголовок, количество ссылок/кнопок/форм, заголовки, текст)
- Heuristic-планировщик выбирает следующее действие на основе цели и уже выполненных шагов
- Если доступна локальная модель — используется model-based reasoning
- **Лимит шагов** (по умолчанию 10, настраивается в UI) предотвращает бесконечные циклы
- При ошибке агент пишет причину и пытается продолжить, а не падает
- Честное завершение с детальным отчётом

### 3. Потоковые обновления в реальном времени (SSE)
- **Server-Sent Events (SSE)** через эндпоинт `/api/events`
- Типы событий: `planning`, `observation`, `reasoning`, `action`, `action_result`, `confirm_request`, `confirm_response`, `warning`, `error`, `blocked`, `completed`, `step_counter`
- Keep-alive ping каждые 30 секунд
- Счётчик подключённых SSE-клиентов в health endpoint
- UI получает обновления мгновенно во время выполнения задачи

### 4. UI с живым журналом и операционной панелью
- **Телеметрия агента** — потоковый журнал с цветовой маркировкой событий, метками времени и типов
- **Индикатор текущей фазы** — иконка, название фазы, номер шага из лимита, прогресс-бар
- **Карточка наблюдения** — разворачиваемая карточка с данными последнего наблюдения (заголовок, URL, ссылки, кнопки, формы, заголовки, текст)
- **Карточка следующего действия** — показывает намерение агента перед выполнением
- **Бейджи режима работы** — видимый режим безопасности (readonly/confirm/full) и источник плана (model/fallback)
- Настраиваемый лимит шагов прямо в интерфейсе
- Версия `v0.2` в хедере

### 5. Полноценный confirm flow
- Для режима «С подтверждением» (confirm) агент ставится на паузу перед `click_link` и `fill_input`
- **AlertDialog** с деталями действия (какое поле, какое значение, какая ссылка)
- Кнопки «Подтвердить» и «Отклонить»
- До ответа пользователя выполнение приостановлено (Promise-based pause)
- Решение логируется в SSE-поток и в базу данных
- Auto-timeout 120 секунд → автоматический отказ
- API эндпоинты: `POST /api/agent/confirm`, `GET /api/agent/confirm/status`

### 6. Визуальное усиление Control Center
- Тёмная тема по умолчанию с контрастными индикаторами фаз
- Цветовая кодировка: cyan для наблюдений, purple для анализа, amber для действий, green для результатов
- Анимация пульсации для активных фаз
- Монолитная шапка с статусами сервера, провайдера, SSE и режима безопасности

## Ограничения

- **Confirm flow** работает только при открытом UI (SSE-подключение необходимо для получения диалога)
- **Heuristic planner** — при отсутствии локальной модели используется простой эвристический выбор действий
- **Один экземпляр браузера** — все задачи используют один Playwright-браузер, параллельное выполнение задач не поддерживается
- **Нет Electron** — проект остаётся веб-приложением
- **Нет мультиагентности** — один агент, один браузер
- **Нет self-healing planner** — при ошибке агент пытается продолжить, но не переключает стратегию
- **Нет облачной синхронизации** — все данные хранятся локально в SQLite
